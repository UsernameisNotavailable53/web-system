# web-system
1. 背景と目的
文化祭本番におけるスパイクアクセス（急激なアクセス集中）を想定した際、Firebase Realtime Databaseの無料枠制限（同時接続数および通信量の上限）に抵触し、システムが停止するリスクが存在していました。
本ブランチ（gemini-ver）では、このインフラ起因のボトルネックを根本から排除するため、バックエンドを完全ローカル環境で稼働させる構成へとアップデートを行いました。

2. 変更内容とメリット
リソース制限の撤廃:
Firebaseへの依存を無くし、自前のNode.jsサーバーによる処理と、ローカルファイル（data.json）へのデータ保存へと切り替えました。これにより、外部サービスの無料枠上限に怯えることなく、無制限かつ安定したシステム運用が可能になります。

Cloudflare Tunnelの継続活用:
通信経路としてはCloudflare Tunnelを使用します。これにより、完全ローカルなサーバーでありながら、従来通りスタッフ端末や本部タブレット等から安全にアクセスすることが可能です。

3. 運用時の必須要件・留意点
今回のアーキテクチャ変更に伴い、システムの起動および運用手順が以下のように変更となります。

実行環境: サーバーとして稼働させるホストPCに Node.js のインストールが必須となります。

プロセスの複数起動: システム稼働中は、ターミナル（コマンドプロンプト等）を2つ開き、以下のプロセスをそれぞれ常時起動しておく必要があります。

プロセスA（サーバー起動）: node server.js

プロセスB（外部公開）: cloudflared tunnel run <トンネル名> 等の既存コマンド

運用上の注意: 稼働中に誤って該当のターミナルを閉じると、システム全体が即時停止します。当日のインフラ運用担当者へは、この点についての周知徹底をお願いいたします。
